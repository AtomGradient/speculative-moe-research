\begin{abstract}
Speculative decoding (SD) accelerates autoregressive LLM inference by having a small \emph{draft} model propose tokens that the larger \emph{target} model verifies in batch. Its effectiveness depends on a high draft acceptance rate---typically achievable when draft and target share the same architecture. For Mixture-of-Experts (MoE) models, which activate only a small fraction of their total parameters per token, two questions arise: \emph{(1) can a tiny dense draft achieve high enough acceptance on an MoE target?}\ and \emph{(2) if not, is there still any speedup?}

We present a systematic study of 306 benchmark runs on \textbf{Qwen3.5-35B-A3B} (35B total, 3B active per token) with 0.8B and 2B dense drafts on an Apple M2~Ultra. We find that cross-architecture acceptance rates are uniformly low ($<$\,4\,\%), yet SD still delivers \textbf{1.18--1.30$\times$ speedup} through a mechanism we call \emph{batch verification amortization}: the target model's single forward pass over $\gamma$ draft positions amortizes the memory-bandwidth cost of loading all 35B parameters. Speedup scales with the target's \emph{total} parameter footprint---not its active count---and increases monotonically with draft length $\gamma$ because batch verification cost grows sub-linearly. A smaller 0.8B draft outperforms a 2B draft by minimizing compute overhead.
\end{abstract}

\section{Introduction}

Speculative decoding~\cite{leviathan2023fast, chen2023accelerating} is now a standard technique for accelerating LLM inference. The recipe is straightforward: a small, fast draft model proposes $\gamma$ candidate tokens; the target model verifies all $\gamma$ candidates in one forward pass; accepted tokens are output, and decoding continues from the first rejection point. When the draft closely matches the target distribution---i.e., the \emph{acceptance rate} $\alpha$ is high---the speedup can be substantial: fewer expensive target forward passes are needed for the same number of output tokens.

Meanwhile, Mixture-of-Experts (MoE) architectures~\cite{shazeer2017outrageously, fedus2022switch} have reshaped the efficiency frontier. A model like Qwen3.5-35B-A3B stores 35 billion parameters but activates only $\sim$3B per token via top-$k$ expert routing. This gives it quality on par with larger dense models at a fraction of the per-token compute. However, \emph{all} expert weights must still reside in memory---and during inference on consumer hardware, loading these weights from memory dominates wall-clock time.

This creates an interesting tension for speculative decoding:

\begin{itemize}[nosep,leftmargin=*]
  \item \textbf{Compute perspective:} MoE is already ``fast'' per token (only 3B active params), so the draft model's compute overhead may negate any benefit.
  \item \textbf{Memory-bandwidth perspective:} MoE must load 35B total parameters from memory per forward pass, creating a large bandwidth bottleneck that batch verification could help amortize.
\end{itemize}

Which perspective dominates? We answer this question empirically with 306 controlled experiments on an Apple M2~Ultra (192\,GB unified memory, 800\,GB/s bandwidth). Our results show that the memory-bandwidth perspective wins: \textbf{SD provides 1.18--1.30$\times$ speedup on MoE despite $<$\,4\,\% acceptance rates}, driven entirely by batch verification amortization of memory bandwidth.

\paragraph{Contributions.}
\begin{enumerate}[nosep,leftmargin=*]
  \item We demonstrate that cross-architecture draft acceptance rates are uniformly low ($<$\,4\,\%) for both MoE and dense targets---a natural consequence of the distributional mismatch between small dense drafts and larger targets from the same model family.
  \item We show that despite near-zero acceptance, SD achieves 1.18--1.30$\times$ speedup on MoE through batch verification amortization---a mechanism distinct from the traditional acceptance-rate-driven speedup.
  \item We establish that SD speedup on MoE scales with \emph{total parameter footprint} (i.e., memory bandwidth demand), not active parameter count, placing MoE between dense-4B and dense-9B in the speedup hierarchy.
  \item We show that SD benefit on MoE is robust across prompt lengths (1.19--1.26$\times$) and increases monotonically with draft length $\gamma$.
\end{enumerate}

\section{Background and Related Work}

\subsection{Speculative Decoding}

In standard autoregressive decoding, each output token requires one full forward pass through the target model. Speculative decoding~\cite{leviathan2023fast, chen2023accelerating} replaces this with a \emph{draft-then-verify} loop:

\begin{enumerate}[nosep]
  \item The draft model $M_d$ autoregressively generates $\gamma$ candidate tokens $\hat{y}_1, \ldots, \hat{y}_\gamma$.
  \item The target model $M_t$ evaluates the prefix $[x, \hat{y}_1, \ldots, \hat{y}_\gamma]$ in a single forward pass, producing logits for all positions.
  \item Tokens are accepted greedily (or via modified rejection sampling) until the first mismatch; a corrected token is sampled at the rejection point.
\end{enumerate}

The classical analysis assumes an acceptance rate $\alpha \in [0,1]$ and a draft-to-target cost ratio $c$. The expected number of tokens produced per verification round is $(1 - \alpha^{\gamma+1})/(1-\alpha)$, giving a theoretical speedup:
\begin{equation}
  S_{\text{classical}} = \frac{1 - \alpha^{\gamma+1}}{(1 - \alpha)(c \cdot \gamma + 1)}
  \label{eq:classical}
\end{equation}

When $\alpha$ is high (e.g., $>$\,70\,\%), this yields large speedups. However, Equation~\ref{eq:classical} assumes that the target's verification cost is independent of $\gamma$---an approximation that breaks down in practice because batch processing of $\gamma$ candidates can be \emph{cheaper} per position than $\gamma$ independent forward passes.

\subsection{Batch Verification Amortization}

On memory-bandwidth-limited hardware, the cost of a single forward pass is dominated by loading model weights from memory. When the target model verifies $\gamma$ candidates in one batch pass, the weights are loaded \emph{once} and reused across all $\gamma$ positions. The marginal cost of adding one more candidate to the batch is primarily compute (matrix multiplication), which is cheap for small batch sizes on modern accelerators.

This means that even when $\alpha \approx 0$ (all drafts rejected), the target model's batch verification of $\gamma$ positions costs less than $\gamma$ sequential forward passes. The ``wasted'' draft evaluations are not truly wasted: they turn sequential, bandwidth-bound operations into a single batched operation. We define the effective speedup under batch amortization as:
\begin{equation}
  S_{\text{batch}} = \frac{\gamma \cdot T_{\text{seq}}}{T_{\text{draft}}(\gamma) + T_{\text{verify}}(\gamma)}
  \label{eq:batch}
\end{equation}
where $T_{\text{seq}}$ is the time for one sequential target forward pass, $T_{\text{draft}}(\gamma)$ is the total draft time, and $T_{\text{verify}}(\gamma)$ is the batch verification time. On bandwidth-limited hardware, $T_{\text{verify}}(\gamma) \ll \gamma \cdot T_{\text{seq}}$ because weight loading is amortized, giving $S_{\text{batch}} > 1$ even when no draft tokens are accepted.

\subsection{Mixture-of-Experts}

MoE models~\cite{shazeer2017outrageously, fedus2022switch} replace the dense feed-forward network (FFN) in each transformer layer with a set of $E$ expert FFN sub-networks and a gating mechanism that selects $k$ experts per token. Qwen3.5-35B-A3B uses 128 experts with top-2 routing, yielding $\sim$3B active parameters per token out of 35B total.

The MoE memory-bandwidth paradox: despite activating only 3B parameters per forward pass, the model must keep all 35B parameters in memory. On unified-memory architectures, this means the full weight matrix must be accessible to the GPU, even though most experts go unused on any given token. The inference speed is thus determined not by the active compute (3B) but by the memory footprint---specifically, by how quickly 20--38\,GB of quantized weights (Q4--Q8) can be streamed through the memory bus.

\subsection{Related Work}

Speculative decoding was independently proposed by Leviathan et al.~\cite{leviathan2023fast} and Chen et al.~\cite{chen2023accelerating}. Subsequent work has improved draft model training~\cite{zhou2024distillspec}, online speculative decoding~\cite{liu2024online}, and multi-token prediction heads for self-drafting. Medusa~\cite{cai2024medusa} adds multiple prediction heads to the target model itself, avoiding a separate draft model.

For MoE inference, prior work has focused on expert offloading~\cite{eliseev2023fast}, expert caching, and routing-aware batching. However, whether SD's batch verification helps amortize MoE's large memory footprint has received little attention. This work addresses that gap.

\section{Analysis and Discussion}

\subsection{Decomposing the Speedup: Bandwidth vs Compute}

Our results cleanly separate two inference bottlenecks:

\begin{itemize}[nosep,leftmargin=*]
  \item \textbf{Memory bandwidth (weight loading)} dominates for large models (Dense~9B, MoE~35B). Loading 10--38\,GB of weights at 800\,GB/s takes 12--48\,ms per pass. Batch verification amortizes this across multiple positions.
  \item \textbf{Compute (matrix multiplication)} dominates for small models (Dense~4B and below). Weight loading is fast, so the GPU spends more time computing, and batch verification provides less relative benefit.
\end{itemize}

The MoE model sits in a unique position: its \emph{compute} profile resembles a 3B dense model (fast), but its \emph{bandwidth} profile resembles a model between 4B and 9B dense (loading 20--38\,GB of expert weights). SD's batch verification targets the bandwidth bottleneck, explaining why MoE gets meaningful speedup despite being ``compute-cheap.''

\subsection{Quantitative Bandwidth Analysis}

We can estimate the effective memory bandwidth utilization. For the MoE Q4 model ($\sim$20.6\,GB weights):
\begin{itemize}[nosep,leftmargin=*]
  \item \textbf{Baseline}: 55.3 tok/s $\Rightarrow$ 18.1\,ms/token $\Rightarrow$ $20.6\,\text{GB} / 0.0181\,\text{s} \approx 1138\,\text{GB/s}$ effective throughput (exceeding the 800\,GB/s theoretical peak, suggesting cache effects and compute overlap).
  \item \textbf{SD $\gamma$=16}: 69.7 tok/s $\Rightarrow$ processes 256 tokens in 3.67\,s. With $\sim$16 verification rounds (most drafts rejected), this is $\sim$229\,ms per round for 16+1 positions, amortizing weight loading across 17 positions per pass.
\end{itemize}

The $\sim$26\,\% speedup at $\gamma$=16 is consistent with loading weights once per round instead of 16--17 times, discounted by draft model overhead and the sub-linear (but not free) cost of batched compute.

\subsection{MoE Q4 vs Q8: Quantization Effects}

The MoE Q8 model achieves \emph{higher} speedup (1.30$\times$) than MoE Q4 (1.26$\times$) at $\gamma$=16. This is expected: Q8 has a larger weight file ($\sim$38.7\,GB vs $\sim$20.6\,GB), making it more bandwidth-bound and thus more amenable to batch verification amortization. The same pattern holds for the dense models: Dense 9B (BF16, $\sim$18\,GB) achieves 2.03$\times$ speedup---the largest in our study---because it is the most bandwidth-constrained target.

\subsection{Practical Implications}

\begin{enumerate}[nosep,leftmargin=*]
  \item \textbf{Always enable SD on MoE models.} With a 0.8B draft and $\gamma=16$, you get a ``free'' 26--30\,\% throughput boost at the cost of $\sim$1\,GB additional memory. There is no quality degradation because SD preserves the target model's output distribution.

  \item \textbf{Use the smallest available draft.} The 0.8B draft outperforms the 2B draft by 0.12--0.14$\times$ speedup. Its compute overhead is lower, leaving more room for the batch verification benefit.

  \item \textbf{Maximize $\gamma$.} Within the range tested ($\gamma \leq 16$), larger $\gamma$ always improved speedup. The sub-linear batch cost suggests that $\gamma = 32$ or higher could yield further gains, limited eventually by KV cache memory and draft model quality.

  \item \textbf{Architecture-matched drafts could unlock much more.} Our acceptance rates are uniformly low ($<$\,4\,\%) because the 0.8B/2B dense drafts don't match MoE's expert routing patterns. A small MoE draft model (e.g., with shared routing) could achieve $>$\,50\,\% acceptance, potentially yielding 3--5$\times$ speedup.
\end{enumerate}

\subsection{Limitations}

\begin{itemize}[nosep,leftmargin=*]
  \item \textbf{Single MoE architecture}: only Qwen3.5-35B-A3B tested. Other MoE models (Mixtral, DeepSeek-V3, etc.) may show different behavior depending on expert count, routing strategy, and parameter distribution.
  \item \textbf{Single hardware platform}: Apple Silicon's unified memory provides a specific bandwidth profile. NVIDIA GPUs with HBM (2--4\,TB/s) or CPU-only inference would show different speedup characteristics.
  \item \textbf{Cross-architecture drafts only}: we did not test MoE-to-MoE drafting. A small MoE draft matching the target's routing could dramatically improve acceptance rates.
  \item \textbf{$\gamma \leq 16$}: larger draft lengths were not tested due to KV cache memory constraints. The monotonically increasing speedup curve suggests potential for higher $\gamma$.
  \item \textbf{Greedy decoding only}: temperature$>$0 sampling may affect acceptance rates and speedup differently.
\end{itemize}

\section{Conclusion}

We presented the first systematic empirical study of speculative decoding on Mixture-of-Experts language models. Our 306-run benchmark on Qwen3.5-35B-A3B (3B active, 35B total) yields four clear findings:

\begin{enumerate}[nosep,leftmargin=*]
  \item \textbf{SD works for MoE}: 1.18--1.30$\times$ speedup despite $<$\,4\,\% acceptance, driven by batch verification amortization rather than draft token acceptance.

  \item \textbf{Total params drive SD benefit}: speedup scales with the target's total parameter footprint (memory bandwidth demand), not its active parameter count. MoE (35B total) benefits more than dense 4B but less than dense 9B.

  \item \textbf{Optimal config = smallest draft + largest $\gamma$}: the 0.8B draft with $\gamma=16$ achieves the best speedup because it minimizes compute overhead while maximizing batch verification amortization.

  \item \textbf{Robust across contexts}: speedup is stable across prompt lengths (1.19--1.26$\times$) and prompt types (factual, code, reasoning, multilingual).
\end{enumerate}

The broader takeaway is that speculative decoding's benefit is not limited to its traditional mechanism (accepted draft tokens reducing target calls). On bandwidth-bound models---including MoE architectures---SD's batch verification itself provides meaningful speedup by amortizing weight loading. This \emph{batch verification amortization} mechanism deserves further study, particularly for increasingly popular MoE models where the gap between active and total parameters continues to grow.
